# 存储节点直连访问与流量模式

本项目的文件下载支持三种访问模式；在线预览为了避免浏览器 CSP/CORS/iframe 限制，统一通过主站受控代理流返回（仍会做登录、权限、路径校验）：

- `PROXY`：网站服务器中转。浏览器访问主站，主站再通过 SFTP/本地文件系统读取文件并返回。最安全，默认推荐。
- `DIRECT`：下载时使用存储服务器直连。主站只做登录、权限、路径校验，然后返回一个短期签名的公开 URL，浏览器直接访问存储服务器。
- `AUTO`：自动模式。下载时节点配置了可用直连基础 URL 则优先直连；否则回退到网站服务器中转。

注意：文件列表里只显示一个“下载”动作，最终走直连还是中转由 VPS/存储节点配置决定；在线浏览/预览固定走主站代理，不在预览页显示直连/中转切换。

## publicBaseUrl 是什么

`publicBaseUrl` 是“存储服务器上对外提供文件访问的 HTTP/HTTPS 基础地址”。

举例：

- 存储节点 SFTP 根目录：`/srv/storage`
- 文件相对路径：`videos/a.mp4`
- 存储服务器公开文件服务：`https://files.example.com/storage/`
- 那么 `publicBaseUrl` 应该填：`https://files.example.com/storage/`
- 浏览器最终访问：`https://files.example.com/storage/videos/a.mp4?expires=...&signature=...`

注意：`publicBaseUrl` 不是 SFTP 地址，也不是主站地址。它必须是浏览器能访问的 HTTP/HTTPS 地址。

## 直连服务器应该怎么配置

最简单但不推荐的方式是在存储服务器上直接开静态目录，例如 Caddy/Nginx 指向 `/srv/storage`。这样虽然能跑，但如果没有签名校验，知道 URL 的人可能长期访问文件。

更安全的推荐方式：

1. 存储服务器部署一个“只负责文件直连”的轻量服务。
2. 服务只暴露指定根目录。
3. 所有请求必须带 `expires` 和 `signature`。
4. 签名由主站根据路径、过期时间、节点 secret 生成。
5. 过期或签名错误直接拒绝。

当前主站已经具备生成短期签名直连 URL 的能力；存储服务器侧还需要部署校验签名的直连服务，或使用支持签名 URL 的对象存储/网关。

## 能不能自动化：浏览时自动部署，用完自动销毁？

可以做，但不建议每次“浏览文件”都启动/销毁服务。原因：

- 首次打开文件会变慢。
- 并发浏览时容易互相影响。
- 频繁改防火墙/进程/证书，故障面更大。
- 如果服务临时启动失败，用户会看到预览失败。

更合适的自动化方式是“按节点启用/禁用直连服务”：

- 在主站存储节点页面点击“启用直连”。
- 主站通过 SSH 在存储服务器执行安装命令。
- 自动安装 Caddy/Nginx 或轻量签名文件服务。
- 自动写入 systemd unit。
- 自动开放指定端口或配置域名。
- 自动回填 `publicBaseUrl`、`directAccessMode=AUTO`。
- 点击“禁用直连”时停止并删除服务。

这比每次预览才部署要稳定得多，也更容易排错。

## 推荐落地方案

### 短期方案：Caddy + 签名校验小服务

适合当前项目继续迭代：

- 主站保留权限判断和签名 URL 生成。
- 存储服务器部署一个 Node/Go 小服务，校验签名后用 `X-Accel`/静态文件流式返回，或者服务自己流式返回文件。
- 主站通过 SSH 一键安装/卸载该服务。
- 优点：改动可控，和当前 SFTP 节点模型匹配。

### 中期方案：MinIO / S3 兼容对象存储

如果主要目标是大文件下载、跨服务器存储，MinIO 更合适：

- 存储服务器部署 MinIO。
- 主站保存 S3 endpoint、bucket、access key。
- 下载/播放大文件时可生成 presigned URL 给浏览器直连；当前在线预览页仍建议通过主站受控代理，避免登录态、CSP、CORS 或 iframe 限制导致预览不可用。
- 浏览器可直接从 MinIO 下载大文件。
- MinIO 原生支持过期签名 URL、权限、对象元数据。

优点是成熟可靠；缺点是现有 SFTP 文件树需要迁移/同步到对象存储，或者同时维护 SFTP 与 S3 两套节点类型。

### 长期方案：对接成熟网盘/对象存储后端

可考虑：

- MinIO：最适合自建对象存储和签名直连。
- SeaweedFS：适合分布式文件系统和大规模小文件。
- Rclone backend：适合接入 WebDAV/S3/OneDrive/Google Drive 等多后端，但权限和预览体验需要额外封装。
- Alist：适合快速聚合多存储，但如果要深度集成现有用户/权限系统，需要二次开发或只作为外部服务。

## 当前建议

当前项目优先级建议：

1. 保留 `PROXY` 作为默认模式，保证安全可用。
2. `AUTO/DIRECT` 只在节点明确配置 `publicBaseUrl` 时启用。
3. 下一步实现“存储节点一键启用直连服务”：通过 SSH 自动安装/配置/卸载存储服务器上的签名文件服务。
4. 如果后续文件量和视频预览需求很大，再新增 MinIO/S3 节点类型，使用对象存储的 presigned URL 替代自研直连服务。
